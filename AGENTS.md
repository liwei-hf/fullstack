# 面试全栈项目协作规范（必须遵守）

> 本文件用于约束 AI 在本仓库中的协作方式、工程决策、代码风格与架构边界。
> 目标是产出一个适合学习、演示、面试讲解，同时具备真实工程组织能力的全栈项目。

## 0. 继承规则

- 继承 `@RTK.md` 终端约束。
- 涉及命令执行时，优先遵循 `rtk` 的代理与元命令约定。
- 如果仓库后续新增更细分的模块级规范，子目录规则可以补充本文件，但不能与本文件冲突。

## 1. 项目定位

你不是简单代码生成器，而是：

- 资深前端工程师
- 全栈工程师
- 系统设计协作者
- 面试项目辅导者

所有输出必须同时满足：

- 可运行：优先保证能启动、能联调、能演示，不做纯展示型 demo。
- 可维护：目录清晰、职责明确、便于后续迭代。
- 可讲解：关键设计需要能讲出原因、取舍和扩展点。
- 可扩展：为后续业务增长、权限体系、AI 能力接入留出边界。
- 不过度设计：优先实现 MVP，不一开始堆砌复杂抽象。

## 2. 默认技术栈

除非用户明确要求替换，否则默认使用以下技术方案：

- Monorepo
- `pnpm workspace + Turbo`
- 管理端：`React + TypeScript + TailwindCSS + shadcn/ui + Radix UI`
- 手机端：`uni-app + TypeScript`
- 后端：`Node.js + NestJS`
- 数据库：`PostgreSQL`
- ORM：`Prisma`
- 鉴权：`JWT`
- API 风格：`RESTful API`
- AI 集成：`Vercel AI SDK`
- AI 高级编排与 RAG 扩展预留：`LangChain`

移动端默认目标：

- `H5` 优先
- 编码时保留后续兼容微信小程序的边界
- 不在初版中为了多端兼容牺牲可读性和开发效率

## 3. 仓库结构约定

默认按以下结构组织：

```text
apps/
  admin/      React 管理端
  mobile/     uni-app 手机端
  server/     NestJS 后端
packages/
  shared/     跨端共享类型、常量、接口契约、工具
```

目录边界必须清晰：

- `apps/admin` 只放管理端页面、组件、hooks、状态管理、接口调用逻辑。
- `apps/mobile` 只放移动端页面、分包、端能力封装、移动端状态逻辑。
- `apps/server` 只放服务端模块、业务逻辑、数据库访问、鉴权和 AI 服务编排。
- `packages/shared` 只放真正跨端共享的内容，例如：
  - `types`
  - `enums`
  - `constants`
  - `api contracts`
  - 通用校验模型或轻量工具

禁止的做法：

- 三端分别复制同一份类型定义。
- 前端页面直接依赖服务端内部实现细节。
- 将 AI 模型调用代码散落在多个业务模块或前端页面中。
- 为了省事把大量无关逻辑堆到单文件。

## 4. 工作方式与需求确认

在开始写代码前，先判断需求是否完整。

如果存在以下情况，必须先澄清：

- 业务目标不明确
- 页面用途不明确
- 用户角色不明确
- 数据结构不明确
- 是否需要数据库不明确
- 是否需要鉴权不明确
- 前后端职责不明确
- 是否需要分页、搜索、筛选、增删改查不明确
- 是否需要文件上传、导出、权限控制不明确

默认执行原则：

- 优先通过阅读现有代码、目录、类型、接口来补齐上下文。
- 能从仓库中确认的信息，不重复向用户提问。
- 如果是高影响决策且无法从上下文得出，再提 1 到 3 个关键问题。
- 如果用户要求“直接做”，则基于默认技术栈和最佳实践先落地 MVP。

## 5. 管理端规范（React）

管理端默认是面试项目中的企业后台。

技术与实现原则：

- 使用 `React + TypeScript + TailwindCSS`
- 基础组件优先使用 `shadcn/ui`
- 可访问性与底层 primitives 优先基于 `Radix UI`
- 保持中后台风格，优先清晰、克制、专业
- 页面逻辑与展示逻辑分离
- 请求层、页面层、业务组件层尽量分层
- 不把表格、筛选、弹窗、表单校验全部塞在一个文件中
- 优先复用现有基础组件，不重复封装同类按钮、弹窗、下拉、表单控件

页面设计要求：

- 页面结构清晰，优先使用标准后台布局
- 必须有合理留白、统一间距、严格对齐
- 避免夸张视觉效果、过多装饰、过深嵌套
- 列表页默认考虑：
  - 搜索区
  - 表格区
  - 分页
  - 加载态
  - 空态
  - 错误态
- 表单页默认考虑：
  - 校验
  - 提交中状态
  - 初始化回填
  - 提交成功或失败反馈

编码要求：

- 统一接口请求封装
- 明确页面组件与通用组件边界
- 样式优先通过 `TailwindCSS` 组织，不引入多套并行样式方案
- 组合 `shadcn/ui` 与 `Radix UI` 时，优先扩展现有组件，不直接复制粘贴大段实现
- Hooks 命名语义化
- 类型定义完整，避免滥用 `any`
- 保证核心函数、重要交互、业务状态切换可以被讲解

## 6. 移动端规范（uni-app）

移动端默认是面试项目中的用户侧应用。

实现原则：

- 使用 `uni-app + TypeScript`
- 默认以 `H5` 为首要运行目标
- 编码时避免把平台差异逻辑直接散落在页面中
- 网络请求、登录态、用户信息、上传能力等优先统一封装
- 不强制使用 `TailwindCSS`、`shadcn/ui`、`Radix UI`
- 可以参考管理端的设计语言、状态反馈和分层思路，但要以 `uni-app` 生态适配为前提

页面与交互要求：

- 页面结构轻量、清晰、以核心流程优先
- 优先保障登录、列表、详情、个人中心、表单提交等基础链路完整
- 所有异步操作必须有反馈
- 页面空态、错误态、提交态要明确
- 如果涉及端能力，优先做兼容层，不把 API 直接写在页面事件里

默认建议：

- 页面目录按业务模块拆分
- 状态管理只解决必要问题，不为简单页面引入过重抽象
- 组件封装保持适度，优先服务复用和可读性
- 不机械照搬 `shadcn/ui` 或 `Radix UI` 的实现方式到 `uni-app` 页面中

## 7. 后端规范（NestJS）

后端默认采用 `NestJS + Prisma + PostgreSQL + JWT`。

模块职责要求：

- `controller` 负责接收请求、参数校验、返回结果
- `service` 负责业务逻辑编排
- `dto` 负责输入输出结构定义与校验表达
- `prisma` 负责数据库访问
- `guard`、`interceptor`、`filter` 负责权限、统一响应、异常处理等横切逻辑

接口设计要求：

- 使用 RESTful 命名
- 列表接口默认支持分页
- 需要时增加搜索、筛选、排序参数
- 返回结构保持统一
- 错误处理保持统一
- 鉴权默认使用 JWT

默认建议优先实现这些基础能力：

- 用户认证
- 用户信息获取
- 列表查询
- 详情查询
- 新增
- 编辑
- 删除
- 基础权限控制

禁止的做法：

- 在 controller 中堆业务逻辑
- 在多个 service 中复制粘贴数据查询逻辑
- 直接把数据库字段结构暴露为前端最终展示模型
- 为了快速完成而忽略异常处理和权限边界

## 8. 共享契约规范

`packages/shared` 用于存放跨端共享契约，目的是降低重复定义与联调成本。

适合放入共享层的内容：

- 业务枚举
- 通用类型
- 统一分页模型
- 通用 API 请求参数或响应结构
- 跨端可复用的常量
- AI 会话消息类型
- 流式事件结构
- 会话元数据
- 知识库检索请求或响应契约

不适合放入共享层的内容：

- 前端专属视图逻辑
- 服务端专属数据库实现
- 与具体框架强耦合的运行时代码

共享层原则：

- 优先共享“契约”，不是共享一切
- 能帮助前后端对齐就共享
- 只在确实跨端复用时再下沉

## 9. AI 集成规范与 LangChain/RAG 预留

本项目默认使用 `Vercel AI SDK` 作为 AI 集成方案，并为后续 `LangChain` 驱动的 RAG 或复杂编排预留清晰边界，避免未来重构。

AI 能力设计原则：

- AI 相关能力必须作为独立领域设计，不允许散落在业务模块内部
- 前端页面不能直接耦合具体模型供应商 SDK 或敏感模型配置
- 模型供应商必须可替换
- Prompt 模板、模型配置、知识库配置必须具备清晰归属
- 手机端可以作为 AI 交互入口，但默认只负责会话 UI、消息流展示和交互体验
- 服务端默认负责模型调用、密钥管理、权限、日志、限流和 RAG 编排

默认集成方式：

- `Vercel AI SDK` 用于聊天、流式响应、会话交互等默认 AI 能力接入
- 手机端和管理端如需 AI 页面，优先基于统一的 AI 接口层或流式事件协议接入
- 服务端中的 AI 能力优先通过独立 `ai` 领域模块暴露，不将模型调用散落在普通业务 service 中

未来引入 `LangChain` 时，默认预留以下边界：

- `provider`：模型供应商适配层
- `prompt`：提示词模板管理
- `chain`：链路编排
- `retriever`：检索器抽象
- `tool`：工具调用抽象
- `memory`：会话记忆或上下文管理
- `trace/log`：调用日志与可观测性
- `config`：模型与 AI 功能配置

如果后续做 RAG，必须分层设计：

- `document ingestion`：文档导入与清洗
- `embedding`：向量化处理
- `retrieval`：检索流程
- `answer generation`：生成回答

禁止的做法：

- 把“导入文档、向量化、检索、生成回答”揉成一个 service
- 在业务模块、页面组件或移动端页面中直接 new 某个模型客户端并发请求
- 把 prompt 文本硬编码在多个 controller 或页面中
- 让具体向量库或模型实现泄漏到上层业务逻辑
- 把 `shadcn/ui`、`Radix UI` 或具体模型 SDK 的实现细节直接搬进 `uni-app` 页面
- 让手机端直接持有敏感模型密钥或供应商配置

推荐的预留方式：

- 在 `apps/server` 中预留独立 `ai` 领域模块
- 手机端 AI 页面通过后端暴露的 AI 接口或流式通道完成交互
- 业务模块如果要接 AI，只依赖抽象后的 AI 服务接口
- 与 AI 相关的配置统一走环境变量和配置模块
- 如果未来需要管理 prompt、知识库和会话记录，优先作为独立子模块演进
- 如果后续需要接入知识库问答、Agent 或复杂工作流，再在服务端引入 `LangChain` 作为 RAG 或编排扩展层

## 10. API 与数据约定

默认遵循以下接口与数据原则：

- 接口命名清晰、语义化
- 分页参数和返回结构统一
- 状态枚举统一管理
- 时间字段命名统一
- 上传、导入、导出等能力要明确边界

面向前端返回数据时：

- 优先返回满足页面需要的稳定结构
- 不直接暴露无意义的底层实现字段
- 在列表模型、详情模型、表单模型之间允许适度区分

## 11. 代码质量规范

所有代码必须满足：

- 命名语义化
- 模块职责单一
- 类型完整

## 16. Git 协作规范

### 分支命名

- `main` / `master`：主分支，随时可部署
- `feature/xxx`：新功能分支，如 `feature/user-auth`
- `bugfix/xxx`：Bug 修复分支，如 `bugfix/login-error`
- `hotfix/xxx`：紧急修复分支，直接从 main 切出
- `refactor/xxx`：重构分支，如 `refactor/api-layer`

### Commit 规范

采用 Conventional Commits：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type 类型**：

- `feat`：新功能
- `fix`：Bug 修复
- `docs`：文档变更
- `style`：代码格式（不影响功能）
- `refactor`：重构
- `test`：测试相关
- `chore`：构建/工具/配置

**示例**：

```
feat(auth): 添加 JWT 登录功能

- 实现用户登录接口
- 生成和验证 JWT token
- 添加 token 刷新机制

Closes #123
```

### PR 流程

- 标题格式：`<type>: <description>`，如 `feat: 添加用户管理模块`
- 描述必须包含：
  - 功能说明
  - 改动范围
  - 测试方式
  - 关联 Issue
- 必须通过 CI 检查
- 至少一人 Code Review

## 17. 环境变量管理

### 文件组织

```bash
.env.example          # 必须提交，包含所有必要变量（无真实值）
.env                  # 本地开发，不提交
.env.development      # 开发环境配置
.env.production       # 生产环境配置
.env.test             # 测试环境配置
```

### 必须遵守的规则

- `.env.example` 必须包含项目启动所需的所有变量
- 真实配置、密钥、Token 禁止提交到 Git
- 后端密钥（数据库密码、API Key、JWT Secret）必须在服务端环境变量
- 前端公开变量以 `VITE_` 或 `NEXT_PUBLIC_` 前缀标识
- 新成员可通过 `.env.example` + 文档快速配置环境

### 变量命名规范

```bash
# 数据库
DATABASE_URL=postgresql://user:pass@localhost:5432/dbname

# JWT
JWT_SECRET=your-secret-key
JWT_EXPIRES_IN=7d

# AI 服务（后端持有）
OPENAI_API_KEY=sk-xxx
ANTHROPIC_API_KEY=sk-ant-xxx

# 应用配置
NODE_ENV=development
PORT=3000
```

### 使用说明

- 新增环境变量时，同步更新 `.env.example`
- 在代码中通过配置模块统一访问，不直接使用 `process.env`
- 启动前检查必要环境变量是否存在

## 18. 错误处理规范

### 后端统一错误格式

所有 API 错误响应遵循统一结构：

```typescript
{
  statusCode: number;      // HTTP 状态码
  code: string;            // 业务错误码
  message: string;         // 错误信息
  details?: unknown;       // 详细错误信息（开发环境）
  path: string;            // 请求路径
  timestamp: string;       // 时间戳
}
```

### 前端错误处理

**请求层封装**：

```typescript
// 统一拦截器处理
axios.interceptors.response.use(
  response => response.data,
  error => {
    const { status, data } = error.response;

    // 401: 跳转到登录
    if (status === 401) {
      clearAuth();
      redirectToLogin();
      return;
    }

    // 403: 无权限提示
    if (status === 403) {
      showToast('无权限访问');
      return;
    }

    // 其他错误：显示错误信息
    showToast(data?.message || '请求失败，请稍后重试');
    return Promise.reject(error);
  }
);
```

**UI 错误反馈**：

- 表单错误：字段级别红字提示
- 提交错误：Toast / Message 提示
- 页面级错误：Error Boundaries + 降级 UI
- 网络错误：显示重试按钮
